The GNU project
Em 1984, um programador brabo chamado Richard Stallman, quem trabalhava no MIT, marcou de criar uma implementação "gratuita" do UNIX. E visão do Stallman era uma moral, e "gratuito" era definido de uma forma legal, ao invés de uma forma financeira, no site deles eles definem como:
"Software livre significa que os usuários do software têm liberdade. (A questão não é sobre o preço.) Nós desenvolvemos o sistema operacional GNU de modo que os usuários possam ter liberdade no uso do computador."

GNU é um acrônimo recursivo para "GNU's is not UNIX", o Stall quis fazer um sistema como o UNIX e que fosse gratuito e do zero, consistindo de kernel e todos os outros programas essências para o sistema, e aconselhou que outros seguissem ele nisso

Umas das grandes consequências da GNU, é o GPL, ou General Public License, a mesma é a personificação da noção de gratuito.
E isso foi extremamente importante! A maior parte dos softwares que tem no Linux e inclusive, o kernel, estão sobre a licença GPL, a licença tem vários regras, mas entre, alguns exemplos, Não transcritos, que eu lembro. são:
    • Todo Software distribuído sobre a licença GPL deve ser código aberto e ter a livre e gratuita permissão para modificação.
    • Todo software que for um derivado ou fork de algum que é da licença GPL, deve também ser da licença GPL.
    • Se o software for distribuído de forma executável, os usuários devem ter a possibilidade de obter e modificar o código fonte.
Dentre outros.
Na época, o equipe da GNU havia feito quase tudo para distribuir um OS completo, exceto o kernel, agora que o Linus faz a sua mágica.

##                          Padronização                     ##
No fim dos anos 80, uma variedade de implementações do UNIX estavam disponíveis, ótimo, mas com um problema, algumas eram baseadas no BSD, outras no System V, e alguns para os 2, sendo assim, cada fabricante/ programador, adicionava oque queria na sua implementação, fazendo ser muito difícil portar programas, oque pressionou a padronização do C.

######                  The C Programming Language             #######

No início dos anos 80, C já teria 10 anos de existência, e foi implementado em uma variedade de dispositivos UNIX e outros sistemas operacionais, com pequenas diferenças entre eles, parte disso porque a linguagem não era padronizada como deveria funcionar.
Esse fatores incentivaram na criação de uma padronização do C, que culminaram em 1989 com a aprovação do ANSI, ou, American National Standars Institute, ou Agência Nacional Americana de Normas/Regras/Padrões.
Que logo foi adotado pela ISO, além de definir a sintaxe e semântica do C e diversas coisas, essa versão é conhecida como C89, ou menos comumente ISOC90, é e completamente descrito no Livro do Ritchie's the C Programming language.
Uma revisão foi adotado pela ISO do padrão em C em 1990, esse padrão é normalmente referido como C99, que contam com diversas modificações na linguagem, essas modificações incluíam a adição do tipo "long long", e de dados Booleanos, comentários com //.
Pointer's ou ponteiros foram restringidos, e arrays de tamanho variável.
Uma coisa ótima sobre o C, é que suas características não são fixas devido ao sistema operacional, ou não apenas fixas no UNIX, qualquer programa em C usando apenas a bibliotecas padrões do C devem ser portáveis através de qualquer distribuição que o suporte.
Isso é uma pura verdade, entretanto, é de suma importância que se note no "Usando apenas as bibliotecas padrões do C", pois, ao avançar na linguagem, e que bibliotecas mais complexas ou funções sejam necessárias, o Windows tem ramificações e diferenciações do C genérico e o do Windows, tal qual o Windows tem código que só podem ser compilados com o compilador próprio do windows, caso contrário não compilaria.

###         Tarefas Executadas pelo Kernel         ###

Entre Diversas coisas, o kernel faz as seguintes tarefas:

• Process shceduling: (Agendamento de Processos em tradução livre) Um computador tem uma ou mais de uma unidade de processamento (CPUs), nas quais executam intruções do programa. Como outros sistemas UNIX o Linux tem uma "Prempitve multitasking operating system" "Sistema operacional de multi-tarefa preventiva" em uma tradução livre. Multi-tarefa significa que multiplos processos (Exemplos são programas) podem estar na memória simultaneamente e cada um talvez receba uso da CPU(s). Preempitve, "Preventivo" significa que a regras governando que faz usso da CPU(s) e por quanto tempo é determinado pelo "Kernel Process Scheduling", (ao invés do proprio programa)

• Memory Managment: (Gerenciamento de Memória): Mesmo que, as memórias de computador são enormes comparado com os tamanhos de anos atrás, o tamanho dos softwares aumento igualmente, então as memórias físicas (RAM) tem recursos limitados no qual o kernel deve gerenciar e dividir a memória entre os processos , de forma igualitária e eficiente. O Linux assim como outros sistemas operacionais modernos usam um sistema de manejamento de memória virtual (Cita secção no futuro), uma técnica que tem 2 vantagens:
    • Processos são isolados uns dos outros, pelo kernel sendo assim um processo não pode interferir na memória de outro, como ler ou escrever.
    • Apenas uma parte do processo precisa ficar na memória ram, sendo assim diminuindo o uso de memória por processo, e permitindo que mais processo fiquem na RAM simultaneamente. Isso leva a um melhor uso da CPU, sendo assim, aumentando a possibilidade de a qualquer momento, tenha pelo menos um processo para que a CPU(s) possa executar.

• Provision of a file system (Provedor de arquivos do sistema): O kernel providencia on sistema de arquivos no disco, permitindo que eles sejam criados, recuperados, atualizados deletados e etc.

• Creation and termination of process (Criação e terminação de processos): O kernel pode carrega um novo programa para a memória e o providenciar com os recursos (Como por exemplo CPU, memória, e acesso à arquivos) que ele precisa para rodar. Um programa quando está em execução se chama processo, e assim que ele completa o kernel garante que ele libere os recursos para que outros programas possam re-utilizar depois.

• Acess to devices (Acesso a dispositivos): Os dispositivos (mouse, monitor, teclado, disk, cds, disquetes e por aí vai) ligados ao seu computador permitem a comunicação entre o computador e o mundo externo permitindo input (entrada) output (saida) ou ambos.
O kernel providencia uma interace padrão para simplificar o acesso à dispositivos, enquanto ao mesmo tempo permite acessos arbitrários de vários processo para cada dispositivo.
• Networking (internet): O kernel recebe e manda menssagens de internet (pacotes) no lugar dos processos do usuário. Essa tarefa inclui rotear a os pacotes de internet para o sistema alvo.

#

• Provision of a system call application programming interface (Provisão de um sistema de chamaods para interfaces de aplicativos programavéis (API)): Processos podem pedir ao kernel diversas coisas usando pontos de entrada conhecidos como System Calls (chamadas de sistema)

Além de tudo que já foi citado, sistemas operacionais multi-usuario como o linux geralmente providenciam usuarios com a abstração do Virtual private computer (Computador virtual privado), ou seja, para cada usuário pode logar no sistema e operar de forma independete dos outros usuários. Por exemplo, cada usuário tem seu próprio espaço no disco (Diretório Home), além de que cada usuário ganha um pedaço da CPU e opera no seu próprio espaço de memória virtual, além de interagir com os dispositivos independentemente e usar a internet. O Kernel naturalmente tenta resolver conflitos entre tentativas de acesso de hardware 


#### Kernel Mode e User Mode ####

O kernel tem dois ambientes principais, a maioria dos processadores atuais suportam que funcionem nesses dois modo, sendo eles: Modo Usuário e Modo Kernel (também chamado de modo supervisor). Instruções de hardware permitem a troca entre um modo para o outro, assim como areas na memória virtual podem ser maracadas como parte do user do space ou do kernel space. Quando rodando no espaço de usuário, a CPU consegue apenas acessar memória no espaço de usuário. E quando rodando no kernel space, a CPU pode acessar ambas.


// Processos versos visão do sistemas //

Em diversas tarefas de programação no dia-dia estamos acustumados a pensar sobre problemas de uma forma focada programação orientada a processos, entretanto devido aos diversos tópicos que serão dissertados nesse guia, é útil para reorientar nossa perspectiva para considerar coisas do ponto de vista do kernel. Para deixar mais claro, agora consideraremos como as coisas são vistas do ponto de vista de um processo e do kernel.

Um sistema funcionando normalmente tem diversos processos, por exemplo coisas acontecendo asincronamente (fora de sincronia). Um processo em execução não sabe quando vai ser a próxima vez que vai ficar "de fora", em que outros processos serão agendados para a CPU (e em que ordem), ou quando vai ser o próximo agendamento (schedule). A entrega de sinais e a ocorrencias de (interprocess) entre processos são manejados pelo kernel, e pode ocorrer a qualuqer momento para um processo.

Um processo normalmente é "burro" sobre a sua estátisticas, por exemplo, ele não sabe onde está localizado na memória ram, e nem mesmo se ele está na ram ou em SWAP (SWAP é um espaço do disco reservado para que o sistema use como memória ram complementar quando necessário). Da mesma forma que um processo não sabe onde que os arquivos que ele acessa estão no disco, ele simplesmente os referencia pelo nome. Um processo opera em isolamneto, ele não consegue se comunicar diretamente com outro processo, um processo por si mesmo não pode criar um novo processo, e nem mesmo terminar o sua própria execução. e um processo não pode se comunicar diretamente com dispositivos de input ou output ligados ao computador.

Do outro lado, "todos" os sistema operacionais tem um kernel, no que controla em tudo e em todos. O kernel facilita a execução de outros processos do sistema. O kernle decide que processo receberá acesso a CPU primeiro, quando ele vai fazer, e por quanto tempo. O kernel mantém estruturas de informação mantendo dados de todos os programas rodando e atualiza essas estruturas quandos novos são criados, quando seus estados são mudados ou encerrados. O kernel mantém todas as estrutura low level que permitem que os nomes do arquivos usados pelos programas sejam traduzidos diretamente para a memória física no disco ou no swap. Todas as comunicações feitas entre os processos são feitas através de mecanismos providenciados pelo kernel. Em resposta chamadas dos programas, o kernle cria e termina processos. E por fim, o kernel, para específico drivers, faz toda a comunicação direta com dispositivos de input e output, transmitindo informação direto do dispostivo para o processo requesitado.


Iremos falar mais para frente no guia/Livro coisas como "Um processo pode criar outro processo", "um processo pode criar um pipe (não conheço uma tradução boa para pipe, cano deve servir)", "um processo pode escrever data em um arquivo", e "um processo pode ser terminado chamando _exit()_.". Não se esqueça que ainda assim, é o kernel quem media todas essas ações, e essas expressões são apenas avbreviações para "um processo pode _pedir para que o kernel_ crie outro processo" e etc.

/////                 Binário                          /////

A base binária é simples, mas para entendermos ela precisamos entender a que já usamos, a decimal, ou seja, na base 10.
Por exemplo, Para escrever o número 721, podemos escrever em 7 Centenas, 2 Dezenas e 1 unidade, ou  70 Dezenas, 20 Dezenas e 1 unidade.
Como todos nós sabemos existem 10 numerais na base 10, de 0 a 9, atenção que começamos a contar do 0, e não existe o numeral 10, é apenas o 1 mais o 0.

Já o binário, funciona apenas com 0 e 1, sendo assim é extremamente rápido, pois lida apenas com "contas simples".

Porém pense que temos apenas valores que podem ser 0 ou 1, O nome da cada valor que pode alternar entre 0 e 1 são chamados de Bits!
Para os representar, vamos usar 1 e 0 e | e X, sendo a barra para 1 e o X para 0, mas qualquer par de algarismos também serve.

Para contar em binário contamos da direita para a esquerda, e não da esquerda para direita igual em decimal.

Imagine que todos estão desligado! Sem atribuirmos valores para eles ainda.

                               ↓
... X   X    X   X  X  X X X X X

Isso mostra que todos os bits estão desligados! Valor 0!
Caso liguemos o primeiro bit, que então terá o valor de 1, qual será o valor total?
1!

E lembre-se de que, o valor do próximo bit é o último vezes dois, sendo assim :

Temos está lista! 
                               ←                                 
... 512 256 128 64 32 16 8 4 2 1 


Como é um sistema binário, as unidades são múltiplas de 2. Para representar números, como 5, você pode o representar como a soma de 1 mais 4!

Ou seja, pense novamente que a | é para ter o valor de 1, ou no caso, ligado!

                           |   |
... 512 256 128 64 32 16 8 4 2 1

Isto representa o número 5! Para o número 6 podemos fazer o 4 e o 2!
Muito simples não é! 

Caso já não tenha percebido, você pode fazer isso com a suas mãos! Ou seja, você pode começar a contar usando a base binária! 
Para fazer isso é simples, considere o dedo abaixado como 0 e o levantado como 1, então você pode fazer igual fez aqui!

/////////             Portabilidade             /////////

Todo compilador de C, tem pelo menos pequenas diferenças, e eles continuam a ser atualizados e melhorados, sendo assim, compiladores como o GCC podem ter suporte completo ao C17 mas não talvez a última versão de C, outros compiladores podem não ter essas versões ou compatibilidade total.

A implementação de atualizações das novas versões de C tendem a demorar bastantes e estar atrás do padrão C atual.

Programas escritos em uma versão específica de C, devem ser compilados apenas a mesma versão que foi escrito, ele pode não compilar e caso compile pode ter um comportamento não previsível,

Mas existem diversos tipos de comportamentos em C, dentre eles :

• Comportamento Definido por implementação.
• Comportamento não especificado.
• Comportamento indefinido.
• Comportamento Local específico.
• Extensões comuns.

Isto é, de acordo com Annex J, do padrão de documentos de C, ao longo da sua aprendizagem da linguagem C, você irá se deparar com todos esse tipos de comportamentos, é muito importante que você conheça todos eles! 


/////////             Definido por implementação             /////////

É um programa em que sua implementação não é bem definida pelo padrão C que pode ter resultados diferentes entre implementações, mas tem um comportamento consistente e documentado.

Um exemplo é o número de bits¹ em um byte²

¹ Bit é a unidade básica de medida de computadores, ele pode valer 0 ou 1.
² Byte é normalmente o nome dado ao agrupamento de 8 bits.

A maioria das implementações são inofensivas, mas podem causar bugs entre implementações, mas sempre que possível evitar escrever este tipo de implementações.
Eu irei dizer quando a implementação for definido por implementação.


/////////                 Comportamento não definido             /////////

Desenvolvedores normalmente confundem comportamento não definido com erros ou falhas ao seguir o padrão de C, mas a intenção de considerar isso é intencional e considerada, esse são comportamentos classificados como não definidos pelo committee de C :

• Dá ao implementador a licença de não pegar um erro no programa quando ele é difícil de ser diagnosticado.
• Evitar definir casos obscuros que iriam beneficiar uma implementação sobre outra.
• Identificar áreas de possível extensão da linguagem em conformidade nas quais o
implementador pode aprimorar a linguagem fornecendo uma definição do
comportamento oficialmente indefinido.

Esses três são bem diferentes mas todos são considerados problemas de portabilidades. Os compiladores (interpretações) tem a possibilidade de fazer as seguintes coisas:
• Ignorar o comportamento completamente e dando resultados imprevisíveis.
• Comportar de uma forma documentada e característica do ambiente com ou sem diagnósticos.
• Terminar a "tradução" ou execução (direcionando o problema).

Todas as opções são ruins, especialmente a primeira, então sempre é bom evitar esse tipo de situação.

/////             Conversão Implícita
A conversão implícita é feita automaticamente pelo compilador quando você assinala um valor de um tipo para outra.

Por exemplo, se você assinalar um valor de um _int_ para um _float_ :

```
// Conversao automatica

```

Como você pode ver, o compilador automaticamente converte o valor de 9 para 9.000000.

Isso pode ser perigoso, já que você pode perder o controle da conversão em certas situações.

Especialmente se for ao contrário, o próximo exemplo automaticamente converte o valor de 9.99 para 9 :

```
// Conversão automática: de float para int

int meu_inteiro = 9.99;

printf("%d", meu_inteiro); // O valor que sairá é de 9
```

Oque aconteceu com o .99? Podemos precisar desse valor! então tome cuidado. É importante ressaltar que você saiba como o compilador irá funcionar nessas situações para evitar resultados inesperados.

Como um outro exemplo, caso você queira dividir dois inteiros, 5 por 2, você sabe que o resultado é 2.5, e como já falamos se você guardar o resultado em um inteiro irá perder o .5, ou seja, é importante que você armazene esse valor em uma variável do tipo _float_ ou _double_.

Exemplo :

```
float divisao = 5 / 2;

printf("%f", divisao); // O valor que irá sair é de 2.000000
```

Mas porque o valor continua sendo 2? e o 0.5? Bem, isso acontece devido ao fato que 5 e 2 ainda são inteiros na divisão. Nesse caso, você precisa manualmente converter esses dígitos de inteiros para números de ponto flutuante. Como ? Assim :


//////                    Conversão Explícita

Conversão explícita é feita manualmente através de parênteses (), antes do valor ser declarado.
Veja como podemos solucionar o problema acima:

´´´
// Conversão manual: de int para float

float divisao = (float) 5 / 2;

```

Você também pode colocar na frente de uma variável, como por exemplo :

```

O valor de divisao é 2.500000, porém caso você queira você pode retirar os zeros extras!

int num1 = 5;

##########                         BSF ou DFS                     #########

Artigos de Fonte: https: //www.codecademy.com/article/bfs-vs-dfs

Existe a Busca em Largura (BFS) e a busca em profundidade (DFS).

BFS é um metódo de busca transversal de camada-por-camada. Começando a partir da raiz do node (ou qualquer outro ponoto no gráfico.), BFS explora imediatamente seus vizinhos, como por exemplo, nodes a distância de 1, então todos os nodes a distância de dois e etc... Esse faz com que ele seja particularmente bom para achar o caminho mais curto entre grafos (graphs) e tree (arvores) não pesados.

A chave da estrura de dados usado no BFS é a lista do espera (FIFO), quem garante que os nodes são explorados na ordem em que foram descobertos.

Algoritomo:
    1. Inicializa uma lista ou um cojunto (ou lista) para manter conta dos nodes visitados.
    2. Começa a explorar as raízes do node e marca como visitado.
    3. Enquanto a lista não estiver vazia:
        •Des-lista o node da frente
        • Processa o node n (ex.: printa e checa se o é o objetivo)
        • emlista todos os vizinhos não visitados (ou "filhos" de uma arvóre) do node n e marcar-los como visitados
    4. Repetir enquanto todos os nodes forem explorados ou um objetivo for encontrado.

``
/* código de exemplo para uma busca em largura em C
 * Este código é usado para exemplificar a lógica 
 * e não uma regra de otimização, mesmo que haja.
 */
 
#include <stdio.h>             // Para permitir usar funções como printf
#include <stdlib.h>            // Permite usar funções como malloc
#include <stdbool.h>           // para usar o tipo booleano




// Estrutura para a lista de adjacência
// Cada vértice do grafo terá uma lista de vizinhos, esaa estrutura representa um elemento (um nó) dessa lista
struct Node {
    int vertex;             // Guarda o numéro do vertice do vizinho
    struct Node* next;      // Ponteiro para o próximo vizinho da lista, como em uma lista encadeada(Cite "Lista Encadeada", X.X.X ?) simples.
    
};



// Estrutura para o Grafo
// Representa o garfo como um todo
struct Graph {
    int numVertices;             // guarda o total de vértice que o grafo possui
    struct Node** adjLists;      // um ponteiro para ponteiro (Na prática, é um array onde cada posição huarda o inicio de uma lista encadeada, uma lista de adjacência)
    bool* visited;                // um array booleano, cada posição i indica se a vetice i já foi visitada pelo algoritimo (true ou false)
};



// Estrutura para a Fila (Queue)
struct Queue {
    int items[100];             // um array estático que suporta 100 elementos na fila
    int front;                  // Guarad o indice do inicio da fila
    int rear;                   // Guarda o indiice do final da fila
};




// Funções e manipulação 

// Cria um nó
struct Node* createNode(int v) {
    struct Node* newNode = malloc(sizeof(struct Node));  // Aloca memória diretamente para um novo nó
    newNode->vertex = v;                                 // Define que esse nó representa o vétice v
    newNode->next = NULL;                                // Como ele acabou de ser criado, ainda não aponta para nenhum nó
    return newNode;                                      // Retorna o primeiro nó criado.
}




// Cria o grafo
struct Graph* createGraph(int vertices) {
    struct Graph* graph = malloc(sizeof(struct Graph));
    graph->numVertices = vertices;
    // aloca memória para estrutura principal  e define a quantidade de vertices
    
    graph->adjLists = malloc(vertices * sizeof(struct Node*));
    graph->visited = malloc(vertices * sizeof(bool));
    
    // aloca memória para o array de listas de adjacencia (uma lista para cade vertice) e para o array de marcação de visitas
    
    for (int i = 0; i < vertices; i++) {
        graph->adjLists[i] = NULL;
        graph->visited[i] = false;
    }
    return graph;
    // um laço de for que inicializa todas as listas como vazias (NULL) e define que nenhum vértice foi visitado ainda (False). No final retorna o grafo pronto
}



// Adicionando arestas (conexões)
void addEdge(struct Graph* graph, int src, int dest) {
    // Adiciona de src para dest
    struct Node* newNode = createNode(dest);
    newNode->next = graph->adjLists[src];
    graph->adjLists[src] = newNode;
    
    // a newMode->... faz o novo nó apontar para o antigo primeiro elemento da lista.
    // graph->adjL... transforma o novo nó no topo da lista.
    
    // Adiciona de dest para src (grafo não direcionado)
    newNode = createNode(src);
    newNode->next = graph->adjLists[dest];
    graph->adjLists[dest] = newNode;
    
    // Como o grafo é _não direcionado_ (se _src_ se conecta a _dest_, então _dest_ também se conecta a _src_), o código se repete e o processo de forma invertida, adicionado _src_ na lista de _dest_
}




// Funções da Fila
struct Queue* createQueue() {
    struct Queue* q = malloc(sizeof(struct Queue));
    q->front = -1;
    q->rear = -1;
    return q;
    // aloca a fila e define -1 para front e rear, indicando que ela está vaiza.
}


bool isEmpty(struct Queue* q) {
    if (q->rear == -1)
        return true;
    else
        return false;
        
    // se o final (rear) for -1, siginifica que nenhum elemento válido está na fila, logo ela está vazia.
}



void enqueue(struct Queue* q, int value) {
    if (q->rear == 99)
        printf("\nA fila está cheia!");
    else {
        if (q->front == -1)
            q->front = 0;
        q->rear++;
        q->items[q->rear] = value;
    }
    
    // Insere um elemento no final da fila. Se for o primeirissimo elemento, ajusta o _front_ para 0, em seguida, avança i índice _rear_ e joga o valor lá dentro.
}



int dequeue(struct Queue* q) {
    int item;
    if (isEmpty(q)) {
        printf("\nA fila está vazia");
        item = -1;
    } else {
        item = q->items[q->front];
        q->front++;
        if (q->front > q->rear) {
            q->front = q->rear = -1;
        }
    }
    return item;
    // remove e retorna o elemnto do inicio da fila (_front_)
    // Avança o _front++_. se o _front_ passar o _rear_, siginific aque a fila esvaziou completamnete, então ambos são reiniciados para -1.
}



// Algoritmo BFS

void bfs(struct Graph* graph, int startVertex) {
    struct Queue* q = createQueue();
    // inicializa o algoritmo criando a fila auxiliar necessaria para o rastreamento.
    
    
    graph->visited[startVertex] = true;
    enqueue(q, startVertex);
    // marca o vértice como visitado e joga na fila para começar a exploração.
    
    
    while (!isEmpty(q)) {
        int currentVertex = dequeue(q);
        printf("Visitado: %d \n", currentVertex);
        // Enquanto houver elementos na fila, o código retira o próximo vértice da vez (_currentVertex_) e o imprime na tela
        
        
        struct Node* temp = graph->adjLists[currentVertex];
        // Pega o primeiro vizinho da lista de adjacencia desse vertcie atual
        
        while (temp != NULL) {
            int adjVertex = temp->vertex;
            // um laço que vai percorrer todos os vizinhos atual. _adjVertex_ armazena quem é o vizinho da iteração atual
            
            
            if (graph->visited[adjVertex] == false) {
                graph->visited[adjVertex] = true;
                enqueue(q, adjVertex);
            }
            // regra de ouro, se o vizingo ainda não foi visitado (false), nós o marcamos imediatamente como visitado (true) e o colocamnos na fila (enque) para que seus proximos vizinhos sejam explorados mais tarde
            
            
            temp = temp->next;
            // avança para o próximo vizinho da lista. Quando acabar os vizinhos (temp == NULL), sai desse laço e volta para o laço da fila.
            
        }
    }
}



int main() {
    struct Graph* graph = createGraph(4);
    addEdge(graph, 0, 1);
    addEdge(graph, 0, 2);
    addEdge(graph, 1, 2);
    addEdge(graph, 2, 3);
    
    // Cria um garfo com 4 vértices (indetificados como 0123) e adiciona 4 arestas, desehando a seguinte estrutura
    // desenhando a seguinte estrutura.
    //· O 0 se conecta a 1 e o 2 .
    //· O 1 se conecta ao 2.
    //· O 2 se conecta ao 3.
    
    
    printf("Iniciando Busca em Largura (a partir do vértice 0):\n");
    bfs(graph, 0);
    return 0;
    
    // chama a função bfs inciando no vetice 0
    // a saida esperada mostrara a ordem da onda de visitação, primeiro o 0, depois os vizinhos diretos deles (1 e 2) e por fim o do vizinho do vizinho (3);
}
